1. 项目概述

1.1 背景与目标

本系统旨在打通聚水潭 ERP 与出口报关业务流程，实现从内部订单号到一整套单证的自动化生成，消除人工填报造成的错误与重复劳动。

核心目标：
- 根据聚水潭内部订单号（支持多选，须同一目的国）自动填制报关草单
- 同步输出 8 类单证，保证"三流一致"（合同流、货物流、资金流）
- 内置合规检查，拦截小规模纳税人风险订单
- 提供灵活的申报合并规则，自动校验海关字符与行数限制

3.1.1 HS编码版本管理

HS编码每年1月1日由WCO协调制度委员会更新，各国海关同步调整。中国海关通常在年初发布新版税则。

版本管理规则：

规则 说明
版本标记 每条商品记录标记 hs_version（如"2022版""2027版"）
废止标记 海关停用的编码 → 系统标记"已废止"，不可用于新草单
存量保留 已生成的旧版草单保留原编码不变，仅标记"编码版本过期"提示
拆分/合并 一个旧编码拆分为多个新编码 → 系统提示受影响商品，需人工确认归属
新编码引入 管理员导入新版税则库后，系统自动列出可用新编码

年度更新流程：

1. 管理员导入新版本HS编码库（Excel/CSV）
2. 系统自动比对：旧版 vs 新版
   ├─ 废止的编码 → 标记 + 列出受影响商品清单
   ├─ 拆分的编码 → 列出候选新编码
   └─ 新增的编码 → 可供新商品备案使用
3. 生成「HS编码变更影响报告」
4. 关务主管逐商品确认调整
5. 确认完毕 → 系统切换默认编码版本

表结构调整：报关商品信息表（3.1）新增字段：
| 字段 | 类型 | 说明 |
|------|------|------|
| hs_version | VARCHAR(20) | 编码版本标识 |
| hs_status | ENUM | ACTIVE / DEPRECATED |

D. 动态表单逻辑：

当用户新增/编辑收发货人时：
  选择「主体类型」= 发货人(出口商·境内)：
    → 仅展示 A通用字段 + B发货人专有字段
    → B区字段全部校验（统一社会信用代码格式、海关注册编码有效性）

  选择「主体类型」= 海外收货人(境外)：
    → 仅展示 A通用字段 + C海外收货人专有字段
    → 国家/地区 与出口管制清单联动校验
    → 是否关联方 默认为「否」，标记为「是」时弹出提示
    → 银行信息非必填但建议填写（生成合同/INVOICE时自动代入）

E. 出口管制联动校验：

海外收货人保存时：
  1. 国家/地区 对照出口管制筛查清单（6.4.1节）
  2. 命中制裁国家 → ⚠️ 红色标记「该国家/地区在制裁清单中，货物可能被限制出口」
     → 状态强制设为「停用」，仅管理员可解除
  3. 企业英文名称 对照 SDN / Entity List 模糊匹配
  4. 命中 → ⚠️ 阻断保存，提示合规部门审查

F. 境外收货人维护界面要求：

  • 列表视图：显示所有收发货人，按「主体类型」分Tab（发货人 / 海外收货人）
  • 国家/地区列：展示国旗图标 + ISO代码 + 中文名称（如 🇺🇸 US 美国）
  • 是否关联方列：是 → 橙色标记 ⚠️，否 → 灰色 —
  • 支持模糊搜索：企业英文名称、国家/地区、联系人
  • 每个海外收货人可关联多个联系人（一对多），但报关单填制时取主联系人

3.2.3 聚水潭客户 → 境外收货人映射

背景：聚水潭订单中的「最终收货人」字段是消费者姓名（如 John Smith），不等同于报关单中的「境外收货人」（法律上的进口商/买方实体，如 ABC Trading Co., Ltd.）。两者概念完全不同，不可自动等同。

A. 概念区分：

维度 聚水潭「最终收货人」 海关「境外收货人」
性质 个人消费者 / 最终签收人 法律上的进口商 / 买方实体
来源 订单Shipping Address收件人 销售合同 Buyer / 商业发票 Consignee
粒度 每个订单可能不同（John / Mary / Bob...） 同一客户多个订单共用同一收货人
报关作用 无直接报关用途 报关单必填项（表头），决定出口管制筛查
字段格式 个人姓名 企业英文全称 + 英文地址

B. 映射逻辑（聚水潭客户 → 境外收货人表）：

生成报关草单时，系统执行以下映射：

  1. 取聚水潭订单「客户名称」字段（如 "ABC Trading Co., Ltd."）

  2. 在收发货人表（主体类型=海外收货人）中模糊匹配：

     a. 精确匹配：聚水潭客户名称 = 收发货人表.企业英文名称 → 直接关联

     b. 模糊匹配（精确匹配失败时）：
        聚水潭客户名称 包含 收发货人表.企业英文名称 的关键词 →
        弹出候选列表（Top 3，按相似度排序），由关务人员确认

     c. 无匹配：
        弹出提示「未找到境外收货人：{客户名称}」
        提供两个选项：
          ┌─ [快速创建] → 打开收发货人新增界面，自动带入客户名称
          └─ [从列表选择] → 打开收发货人选择弹窗，人工指定

  3. 匹配成功 → 境外收货人信息自动填入报关草单表头第29项

  4. 同一客户 = 同一境外收货人（一经确认，后续不再重复询问）

C. 模糊匹配算法：

匹配策略 权重 说明
英文名称精确匹配 100 完全相同 → 自动关联
英文名称包含匹配 80 聚水潭客户名包含收发货人英文名
关键字分词匹配 60 提取聚水潭客户名的词（排除 Ltd/Inc/Co 等后缀），与收发货人英文名分词对比
联系人姓名匹配 40 聚水潭客户名称 = 收发货人表.联系人
历史关联记录 90 同一聚水潭客户ID此前已确认过映射 → 直接复用

D. 映射关系存储（客户-收货人关联表）：

CREATE TABLE customer_consignee_map (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    jst_customer_id VARCHAR(50) NOT NULL,       -- 聚水潭客户ID
    jst_customer_name VARCHAR(200),             -- 聚水潭客户名称（快照）
    consignee_id    BIGINT NOT NULL,            -- 收发货人表ID
    match_method    ENUM('EXACT','FUZZY','MANUAL'),  -- 匹配方式
    last_matched_at DATETIME,                   -- 最近一次匹配时间
    created_by      VARCHAR(50),
    created_at      DATETIME,
    UNIQUE KEY uk_customer (jst_customer_id)
);

E. 校验规则：

校验项 规则 失败处理

4.5.1 为什么需要手动确认

聚水潭「标记|多标签」字段存在以下不可靠因素：

情况 影响
标记字段为空 聚水潭操作人员未维护，无法判断运输方式
标记字段错误 实际走海运但标记为「海外-空运」，反之亦然
标记非运输相关 标记可能用于其他业务目的，与运输方式无关

因此标记仅作筛选参考，不可自动带入运输方式。

4.5.3 业务规则

  • 运输方式为批次级设置：同一批次所有订单共用同一运输方式
  • 确认后的运输方式写入该批次报关草单的「运输方式」字段
  • 运输方式一经确认，后续报关单证（报关单、PACKING LIST、INVOICE 等）统一引用此值

5. 申报合并规则引擎

5.1 合并前提条件（硬性约束）

合并申报要求同时满足：
1. 相同供应商（supplier_code 相同）
2. 相同原产地（origin 相同）

违反任一条件，即使HS编码相同也不允许合并到同一申报行。

5.2 合并策略选项（6+1种）

策略代码 合并维度 适用场景
MERGE_HS 相同HS编码 同类商品最大化合并，申报行数最少
MERGE_HS_NAME HS编码 + 品名 同HS但不同品名需区分时
MERGE_HS_NAME_SKU HS编码 + 品名 + 商品编码 需保留SKU粒度
MERGE_HS_NAME_STYLE HS编码 + 品名 + 款式编码 女包/鞋服按款式区分
MERGE_HS_NAME_STYLE_MATERIAL HS编码 + 品名 + 款式编码 + 材质 最精细，申报要素完整
MERGE_HS_NAME_MATERIAL HS编码 + 品名 + 材质 款式不重要但材质需区分
NO_MERGE 不合并，每条SKU独立一行 申报行数最多，精确一致

5.3 校验规则

5.3.1 申报要素字符数校验（250字符限制）

申报要素字段（如款号列表）合并后总长度 > 250字符 → 报错

处理选项（用户可选）：
- 截断 + 加"等"字：保留前N款，末尾追加「等」字（如：A001、A002、A003等）
- 切换合并策略：提示用户选择更细粒度的合并方式
- 手动编辑：允许用户在生成前手动缩写申报要素

⚠️ 注意："截断+等"方案在法规上无明文禁止，但存在口岸执行尺度差异。详见第13章《申报要素截断合规说明》。

5.3.2 申报行数校验

海关出口报关单最大申报行数：50行（标准H2000申报格式）
生成申报行数 > 50行 → 报错，提示切换合并策略

处理选项：
- 提示当前行数 vs 上限
- 推荐可压缩到50行以内的合并策略
- 拆分为多张报关单（需手动确认）

5.4 合并规则执行流程

输入订单列表
    │
    ▼
过滤：已审批商品信息 & 已通过合规检查
    │
    ▼
按（供应商 + 原产地）分组
    │
    ▼
在每组内，按选定合并策略归并行
    │
    ▼
校验：字符数 ≤ 250 & 行数 ≤ 50
    │
    ├── 通过 → 生成草单
    └── 失败 → 报错提示，要求调整策略

6. 合规检查模块

6.1 小规模纳税人逻辑

出口订单涉及小规模纳税人供应商时，按以下逻辑处理：

if 供应商.纳税人类型 == "小规模纳税人":
    if 供应商.合规状态 == "不合规":
        → 硬性拦截，禁止生成报关单
        → 界面显示：【禁止操作】该供应商（XX）已标记为不合规，
                     不允许进行单证出口。请联系合规部门处理。

    elif 供应商.合规状态 in ("合规", "待确认"):
        → 弹出二次确认对话框
        → 提示："该批次包含小规模纳税人商品，单证出口将无法退税，
                  且存在合规风险。确认继续？"
        → 用户需手动勾选确认复选框后方可继续
        → 注意：待确认(PENDING)状态下 can_doc_export 仍为 TRUE，
          允许出口，但需关务主管确认风险

6.2 合规检查项汇总

检查项 触发条件 处理方式
小规模+不合规 订单含不合规小规模供应商 硬性拦截
小规模+合规/待确认 订单含小规模供应商（合规） 二次确认弹窗
商品未审批 报关商品状态≠已审批 警告，不可生成
HS编码缺失 商品无有效HS编码 错误提示
目的国冲突 多选订单目的国不一致 实时阻止勾选

6.3 双人复核机制（AEO认证要求）

依据海关AEO认证标准，关键业务节点需实施双人复核。

操作节点 复核类型 执行方式
报关草单提交审核 强制复核 制单人提交 → 关务主管审核通过 → 方可进入「已确认」状态
报关草单修改 强制复核 修改后须复核人确认，修改记录永久保留
单证作废/删除 强制复核 不可单人操作，须双人确认
申报价公式变更 强制复核 公式修改后需关务主管审批生效
HS编码归属变更 强制复核 商品HS编码调整须双人确认
供应商合规状态变更 强制复核 从PENDING→COMPLIANT或NON_COMPLIANT须双人确认

复核界面：
- 复核人可见操作人、操作内容、变更前后对比
- 复核通过 → 记录复核人+时间戳
- 复核驳回 → 回退至操作人并附驳回原因

6.4 出口管制与品牌授权检查

6.4.1 出口管制筛查

检查项 触发条件 处理方式

合规结论（全网查证）

海关对装箱单与报关单的一致性要求不是逐行死板对应，而是遵循逻辑闭合原则：

层面 要求 本系统实现

合并显示清单说明

为什么需要合并显示清单？

在实际报关场景中，海关审核 PACKING LIST 时看的不是逐箱明细，而是按品名归集后的汇总视图。逐箱明细是仓库/货代核对装柜用的；合并显示清单才是海关比对报关单的核心参照。

层次 内容 受众
逐箱明细 每个纸箱的SKU/数量/重量 仓库人员、货代装柜核对
合并显示清单 按HS归集，品名+总量+箱数统计 海关审核、报关行比对
总计行 整票货的总箱数/总毛重/总净重/总体积 提单、舱单对照

合并规则：

  • 归集键：HS编码 + 商品名称(name_en) — 同一HS编码下的不同SKU显示为独立行
  • 数量合计：同一归集键下所有箱中的数量求和
  • 涉及箱数：该品名出现在多少个不同的箱号中（混装箱会被计数多次）
  • 毛重/净重合计：同一归集键下各箱的毛重/净重求和
  • 体积合计：箱规 × 涉及箱数（同品名同箱规时）；不同箱规时按实际逐箱累加
  • 合并显示清单合计行：所有归集行的数量/毛重/净重/体积求和

混装箱在合并显示清单中的处理：

示例：Box 5（混装箱）含 SKU-A ×3 + SKU-B ×4

  SKU-A      总数量  3  ← 从Box 5拆出    涉及箱数  1（Box 5）
  SKU-B      总数量  4  ← 从Box 5拆出    涉及箱数  1（Box 5）

  注意：合并显示清单的毛重/净重/体积是逐箱分拆的估计值，
  实际混装箱内各SKU的重量按数量比例分摊：
    SKU-A分摊毛重 = 20.0 × 3/(3+4) ≈ 8.57
    SKU-B分摊毛重 = 20.0 × 4/(3+4) ≈ 11.43

混装箱重量分摊精度说明：

混装箱各SKU的毛重/净重按数量比例分摊，为估计值：
  - 分摊公式：SKU_i分摊毛重 = 整箱毛重 × (SKU_i数量 / 箱内总数量)
  - 允差范围：分项加总 = 箱总重（数学等式必成立），但各项真实值 ±3% 内可接受
  - 注意：如果各SKU单位重量差异很大（如鞋子 vs 配饰），
    比例分摊误差会增大，建议尽量不分装重量差异大的SKU混装

核心原则：PACKING LIST 三层结构——逐箱明细（仓库核实） → 合并显示清单（海关审核） → 总计行（单证对照），行级不必对应（箱级≠HS级），但总量闭合必须成立。海关法要求的是"单证之间不得矛盾"，不是"单证之间格式相同"。

13. 申报要素截断合规说明

13.1 法规依据

  1. 《中华人民共和国海关进出口货物报关单填制规范》（海关总署公告2019年第18号）规定：
    - 申报要素应完整、准确填写，不得遗漏。
    - 商品名称、规格型号等字段应与实际货物一致。

  2. 《海关进出口货物申报管理规定》（海关总署令第103号）规定：
    - 收发货人或其代理人应按照海关规定的格式和要求如实申报。
    - 申报内容应当真实、准确、完整。

13.2 250字符限制的来源

  • 海关H2000系统对「商品规格型号」字段的技术限制为250个中文字符（含标点）。
  • 此限制为系统字段长度约束，非法规明文规定。
  • 超长会导致系统无法接收申报数据。

13.3 「截断+等字」方案的合规风险分析

风险等级 情形 海关可能处置
低风险 款号较多，截断后加「等」字，保留部分已能唯一识别商品 通常接受，前提是品名+HS编码已能识别
中风险 截断部分包含材质、成分等关键申报要素 可能要求补充申报，或影响归类认定
高风险 截断后导致申报要素与实物严重不符，或故意隐瞒关键信息 涉嫌申报不实，可能面临行政处罚（海关法第86条）

13.4 建议操作规范

  1. 优先采用更细粒度合并策略，避免触发250字符限制。
  2. 如确需截断，应满足：
    - 保留部分已能完整描述商品关键特征（品名、材质、用途）
    - 「等」字仅用于表示「同款其他颜色/尺码」，不得用于省略不同款号
  3. 在系统操作日志中保留完整款号清单，以备海关核查时提供补充材料。
  4. 建议在首次采用此方案时，提前与申报口岸海关沟通确认。

13.5 结论

「截断+等字」方案在法规上无明文禁止，但存在口岸执行尺度差异。系统应保留此方案作为应急选项，但需：
- 在界面显著位置提示风险
- 要求关务人员确认后方可执行
- 保留完整申报要素（未截断版）备查

14. 申报行数50项全国统一查证

14.1 查证方法

对以下三个来源进行了交叉验证：

来源 查证内容 结论
《报关单填制规范》（海关总署公告2019年第18号）全文 逐条检索「50」「五十项」「上限」 无明文规定 — 填制规范不限制申报项数
行业实务资料（thecustoms.com.cn 等） 2016年改版报道 单份报关单从20项提升至50项，属「重大改版」
中国国际贸易单一窗口 系统架构 全国统一平台，一点接入、全国通关

14.2 核心结论

50项限制是H2000/H2010通关管理系统的技术参数，非法规条文。

推理链：
1. 《报关单填制规范》中没有对申报项数做任何数量限制
2. 50项来自H2000系统数据库设计时的字段分配（每份报关单最多存储50条商品记录）
3. 中国国际贸易单一窗口是全国统一平台（「一点接入、全国通关」），所有口岸使用同一套系统后端
4. 因此，50项限制对所有口岸统一适用，不存在口岸差异

14.3 法规依据（间接）

15. 拆单逻辑推荐方案

15.1 设计原则

当合并后申报行数超过50项时，系统需自动拆分为多份报关单。推荐 「自动推荐 + 人工确认」 双轨模式。

15.2 自动推荐拆单优先级

按以下优先级依次拆分，直至每份报关单 ≤50项：

优先级 拆分维度 说明 合规依据
P1 供应商 同一供应商的商品放在同一份报关单中 供应商不一致无法合并申报（发票流一致）
P2 最终目的国/目的港 同一目的港放在同一份 报关单「运抵国」字段唯一
P3 HS编码章类（2位） 同章类商品归入同一份 便于海关归类审核
P4 HS编码（完整） 同HS编码尽量不拆分 避免同一编码出现在两份报关单中

15.4 人工调整方式

用户可以对系统推荐的拆分结果进行以下调整：

  • 手动重新选择合并策略：回到合并规则引擎，换一种合并方式重新生成
  • 手动拖拽调整：在拆分结果界面拖拽商品行到不同报关单
  • 手动拆单：直接指定哪些内部订单号归入第N份报关单

15.5 硬性约束

  • 每份报关单商品项数 ≤ 50项（H2000系统强制）
  • 每份报关单「申报要素（规格型号）」总字符数 → 系统不设统一上限，但单行 ≤ 250字符
  • 同一份报关单内供应商必须一致、原产地必须一致

16.2 推荐：本地私有化部署

推荐理由（按重要性排序）：

  1. 数据敏感度极高：报关数据包含：供应商进价（商业机密）、客户售价、退税率、HS编码归类和申报要素。一旦泄露可直接暴露整个供应链成本和利润结构。
  2. 税务合规要求：退税申报涉及三流一致验证，数据链路需要可审计。本地部署便于保留完整操作日志，满足税务局核查要求。
  3. 长期成本更低：按5年周期计算，本地部署比SaaS节省约40~50%（15~35万 vs 25~75万）。
  4. 可与现有系统集成：聚水潭API已打通、电子签系统（腾讯电子签）已有，本地部署可直接对接内网环境。
  5. 不受外部服务中断影响：单一窗口系统本身是外部依赖，若报关系统再依赖云端SaaS，双重外部依赖风险叠加。
